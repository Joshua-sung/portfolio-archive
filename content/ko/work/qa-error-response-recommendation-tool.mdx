## 상황 / 과제

검수 업무에서는 반복되는 시스템 오류를 작업자에게 이해하기 쉬운 말로 설명해야 하는 상황이 많았습니다. 기존 방식은 검수자가 오류 원문을 직접 해석하고 매번 답변을 새로 작성해야 했기 때문에, 같은 유형의 문제도 사람마다 설명 방식이 달라질 수 있었습니다.

비개발자 검수자 입장에서는 시스템 오류를 정확히 이해하기 어렵고, 작업자 입장에서는 반려 사유가 일관되지 않게 전달될 위험이 있었습니다. 과제는 반복 검수 지식을 검색 가능한 운영 자산으로 바꾸는 것이었습니다.

## 실행

이 문제를 개인 역량의 문제가 아니라 운영 지식이 구조화되어 있지 않은 문제로 봤습니다. 먼저 도구 구축용 원본 처리이력 2,438건을 분석해 에러, 검수 코멘트, 시스템 이벤트를 분류했습니다.

이후 에러와 코멘트 페어 608건을 정리하고, 표준 케이스 24개, 답변 템플릿 48개, 검색 인덱스 48개로 구조화했습니다. 과거 코멘트를 그대로 복사하는 방식이 아니라, 작업자에게 바로 보낼 수 있는 상세 안내와 짧은 안내 템플릿으로 나누어 실제 검수 흐름에 맞게 정리했습니다.

도구는 서버나 별도 설치 없이 사용할 수 있도록 단일 HTML 형태로 만들었습니다. 검수자가 에러 원문을 붙여넣으면 유사 케이스와 작업자용 반려 문구를 확인할 수 있도록 했고, 표준 케이스가 없는 경우에는 새 패턴을 쉽게 기록할 수 있게 만들어 지식 DB가 계속 보강되도록 설계했습니다.

## 결과

성과는 개발자 제공 전체 검수이력 43,846건과 전체 운영자 수동 처리 이벤트 15,873건을 기준으로 다시 확인했습니다. 도구 도입 후 제 1인 1업무일 평균 반려 처리량은 9.81건이었고, 도입 전 전체 비교군의 5.71건 대비 +71.9%, 약 1.72배 높은 처리량을 기록했습니다.

전체 수동 검수 처리 기준으로는 31.44건에서 132.40건으로 약 4.21배 증가한 처리량이 확인됐습니다. 반복 검수 지식을 개인 기억에서 꺼내 검색 가능한 운영 도구로 바꾼 것이 핵심 성과였습니다.

## KPI 판단

이 도구는 직접 매출이나 검증된 비용 절감액이 없기 때문에 운영효율 지표 중 가장 강한 수치를 대표 KPI로 선택했습니다. 전체 수동 검수 처리량 4.21배 증가를 전면에 두고, 반려 처리량 1.72배 개선은 실제 반려 응답 업무와 더 직접적으로 연결된 보조 KPI로 배치했습니다. 43,846건의 전체 검수이력은 성과 비교의 신뢰도를 뒷받침하는 근거입니다.

## 도구와 기술

- 과거 검수 처리이력 분석
- 에러와 검수 코멘트 분류 체계
- 표준 케이스와 답변 템플릿 라이브러리
- 유사 케이스 검색 인덱스
- 서버 없는 단일 HTML 배포 방식

## 협업

검수자, 비개발 운영자, 개발자 제공 검수 이력이 연결된 작업이었습니다. 저는 시스템 수준의 에러 흔적을 작업자에게 전달 가능한 문장으로 번역하고, 검수자가 안전하게 재사용할 수 있는 형태로 구조화했습니다.

## 배운 점

운영 지식은 검색 가능하고 재사용 가능하며 업무 중 보강하기 쉬울 때 확장됩니다. 검수 운영에서는 작은 로컬 도구라도 반복 판단의 근거를 담으면 처리 속도와 설명 품질을 함께 개선할 수 있습니다.
